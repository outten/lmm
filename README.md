# outten/lmm

> A Large Math Model — fine-tuned to write code (Python + Mermaid) for every math question, with a companion runtime that executes the code, renders charts, and gives you back the code so you can re-run it forever.

This is not a math-flavored chatbot. It's a model whose output is **executable**: every numeric claim is backed by code, every chart is generated by code, every diagram is Mermaid — and the code is yours to keep.

## Quickstart

If you already have Ollama and Python 3.9+:

```bash
git clone https://github.com/outten/lmm
cd ollama-lmm

# 1. Install runtime deps (one-time)
make install

# 2. Build the model
make build

# 3. Chat — with code execution
make run
```

Or run a one-shot question:

```bash
python3 sandbox/ollama_cli.py "Correlate SPY and QQQ daily closes in 2024."
```

You'll get a response that includes Python code. The runtime extracts the code, executes it in a sandbox, captures the output, and prints the result alongside.

## How it works

```
   ┌─────────────────────┐
   │   Your question     │
   └──────────┬──────────┘
              ▼
   ┌─────────────────────┐
   │  outten/lmm model   │  ← fine-tuned for the code-first format
   │   (Ollama)          │
   └──────────┬──────────┘
              ▼ "```python\n... ```mermaid\n..."
   ┌─────────────────────┐
   │   sandbox runtime   │  ← extracts code blocks
   │  - python subprocess│     executes Python
   │  - mermaid-cli (mmdc)│    renders Mermaid to SVG
   └──────────┬──────────┘
              ▼ charts.png, answer.svg, stdout
   ┌─────────────────────┐
   │   Your terminal     │  ← code + output + (optionally) summary turn
   └─────────────────────┘
```

The **code-first doctrine** (see [PROPOSAL.md §13](PROPOSAL.md#13-code-first-doctrine)) is the defining design principle. In short:

  - Computers compute. LLMs hallucinate. The math comes from code execution, not from the model.
  - Every response contains a fenced code block (Python and/or Mermaid).
  - You get the code so you can re-run it anytime — no LLM, no tokens, deterministic.

## Run the eval harness

```bash
# Tiny smoke set (no GPU needed)
make eval

# Full GSM8K + MATH (after `make download-benchmarks`)
make eval-gsm8k
```

Eval is **code-verified**: the model's prose is never string-matched. The harness extracts the code, runs it in the sandbox, and compares the numeric output to ground truth. This is the metric that actually matters — see [PROPOSAL.md §6](PROPOSAL.md#6-evaluation--benchmarking).

## Run the test suite

```bash
make sandbox-test
```

12 tests cover code extraction, the Python sandbox (sympy, matplotlib, timeouts, exit codes), and the Mermaid renderer.

## Training a new version

See [PROPOSAL.md §5](PROPOSAL.md#5-fine-tuning-methodology) for the full plan, and [docs/CLOUD_GPU.md](docs/CLOUD_GPU.md) for cloud GPU pricing, provider comparison, and a per-version recommendation.

Short version:

```bash
# 1. Reformat a public dataset into the code-first format
python3 training/scripts/reformat_dataset.py \
    /path/to/metamath.jsonl \
    -o training/data/sft.jsonl \
    --format metamath --limit 50000

# 2. Fine-tune (requires GPU)
make train TAG=v0.1

# 3. Convert to GGUF and rebuild
make convert TAG=v0.1
make build TAG=v0.1

# 4. Eval
make eval TAG=v0.1
```
